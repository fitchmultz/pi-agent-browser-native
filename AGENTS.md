# pi-agent-browser-native

Native `pi` integration of `agent-browser` as a `pi` tool.

## Product-specific guidance

- Do **not** bundle `agent-browser` in this project.
- Assume `agent-browser` is installed separately and available on `PATH`.
- Target the current locally installed `agent-browser` version only.
- Do **not** add backwards-compatibility shims for older upstream versions.
- Keep the integration thin and close to upstream `agent-browser` behavior.
- The primary UX is the agent invoking the native tool directly, not a slash-command-heavy manual workflow.
- Do **not** overengineer or solve hypothetical problems that do not exist in observed behavior.
- Do **not** add a reusable named "browser recipe" runtime layer on top of native `job` / `qa` / `batch` without new benchmark or dogfood evidence and an explicit design pass; the current no-adopt decision and revisit criteria live in [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md#no-reusable-recipe-layer-yet) and [`docs/SUPPORT_MATRIX.md`](docs/SUPPORT_MATRIX.md).
- Thoroughly check official `pi` docs/examples/source behavior before inventing bespoke integration patterns. Prefer an official `pi` mechanism whenever one exists.
- Prioritize the global install path first. Most users will install this extension globally, not as a project-local extension.
- Keep the local extension-side documentation good enough that an agent can use the tool without relying on direct `agent-browser` binary help; when upstream `agent-browser` changes, update the repo-readable command reference, prompt guidance, README/docs, and any relevant tests in the same work.
- For this repository, assume a single operator model: no human and no other agent is making changes here besides you.
- Treat every lingering scratch file, temp artifact, browser session, tmux session, or other side effect related to this repository as your responsibility to clean up.

## Documentation placement

- Put user-facing product docs in `README.md` and `docs/`.
- Put agent-specific operational notes, workflows, and testing procedures in this `AGENTS.md`.
- Write documents as complete documents, not iterative logs, unless the document is explicitly meant to be iterative such as `CHANGELOG.md`.

## Upstream capability baseline and command reference

When upstream `agent-browser` changes version or help text, keep three layers aligned:

1. **Canonical metadata** — `scripts/agent-browser-capability-baseline.mjs` defines the targeted upstream version, which `agent-browser <args>` help outputs are sampled for drift checks, and which literal tokens must appear in upstream help and in human-written sections of `docs/COMMAND_REFERENCE.md` (grouped as inventory sections). Nothing in this file shells out to `agent-browser`; rebaselining is an explicit maintainer edit after comparing real `--help` output from the installed binary.

2. **Human command guide** — `docs/COMMAND_REFERENCE.md` holds the readable workflows, examples, and constraints. Outside the two HTML-comment bounded generated regions (`upstream-baseline`, `capability-token-baseline`), edits are normal prose. Every human-authored token listed in the baseline must appear somewhere in the doc body so agents searching the repo see real usage, not only generated lists.

3. **Generated blocks** — The baseline checker renders versioned boilerplate into `docs/COMMAND_REFERENCE.md`. After changing the baseline metadata, run `npm run docs -- command-reference write`. Do not hand-edit inside the `<!-- agent-browser-capability-baseline:start ... -->` / `<!-- ...:end ... -->` markers.

Release-readiness checklist: [`docs/SUPPORT_MATRIX.md`](docs/SUPPORT_MATRIX.md) maps each baseline inventory section to human docs, wrapper/runtime behavior, tests, and verification gates. Refresh it when rebaselining upstream.

Verification stack:

- `npm run typecheck` — runs the same `tsc --noEmit` step as the default verify gate (`scripts/project.mjs` `verify typecheck`) without playbook drift, unit tests, or live upstream command-reference sampling; use for fast TypeScript iteration.
- `npm run docs` (or `npm run docs -- command-reference check`) — generated blocks match `agent-browser-capability-baseline.mjs`.
- `npm run verify -- command-reference` — the above plus live `agent-browser --version` and help sampling against the same baseline (requires the targeted upstream on `PATH`).
- Unit coverage for the verifier and block renderer lives in `test/verify-command-reference.test.ts`.

**npm verification facade:** `npm run verify` and `npm run docs` delegate to [`scripts/project.mjs`](scripts/project.mjs) (`verifySteps` / `docsSteps`). The default gate runs generated playbook drift, `tsc --noEmit`, `tsx --test --test-concurrency=1 test/**/*.test.ts`, command-reference generated-block checks, then live `scripts/verify-command-reference.mjs` (needs the baseline-matching `agent-browser` on `PATH`). `release` concatenates the default steps with `verify-package.mjs --smoke-pi`, then the Crabbox platform smoke doctor and full macOS/Ubuntu/native-Windows matrix. `package.json` `prepublishOnly` runs `npm run verify -- release` and then `npm pack --dry-run` during `npm publish` (unless publish uses `--ignore-scripts`). [`test/project-verify.test.ts`](test/project-verify.test.ts) locks default, `release`, `real-upstream`, `dogfood`, `platform-target`, `platform-smoke`, `package-pi`, and combined-docs orchestration so those steps cannot be dropped silently. `npm run verify -- dogfood` runs `scripts/verify-agent-browser-dogfood.ts` against the real `agent-browser` on `PATH` and public `example.com`; it complements but does not replace interactive tmux dogfood. Run `npm run verify -- --help` for opt-in modes and supported passthrough flags.

**Crabbox platform smoke:** [`docs/platform-smoke.md`](docs/platform-smoke.md) is the source of truth. The release-blocking gate is `npm run check:platform-smoke`, `npm run smoke:platform:ubuntu-image`, then `npm run smoke:platform:all` (the full matrix is also embedded in `npm run verify -- release`). It must pass on `macos`, `ubuntu`, and `windows-native`; missing Docker, macOS SSH, Parallels, Windows template/snapshot, upstream `agent-browser`, or browser dependencies is a blocked setup, not a skip. The `platform-build` suite proves `npm ci`, `npm run verify -- platform-target`, `npm pack`, clean project `npm install --no-save`, `pi install -l ./node_modules/pi-agent-browser-native`, and `pi list` without `pi -e .`; `browser-dogfood-smoke` proves the native wrapper against a real browser on each target. Artifacts live under `.artifacts/platform-smoke/` and cleanup failures fail the target.

Release-oriented notes also live in [`docs/RELEASE.md`](docs/RELEASE.md) under pre-release and real-upstream sections.

**Opt-in real upstream contract:** `npm run verify -- real-upstream` runs `test/agent-browser.real-upstream-contract.test.ts` with `PI_AGENT_BROWSER_REAL_UPSTREAM=1` against the real `agent-browser` on `PATH` (must match `scripts/agent-browser-capability-baseline.mjs`). Besides inspection, skills, `open`/`eval --stdin`/`snapshot -i`/`batch`, and managed-session continuity, the suite exercises a representative core command matrix on localhost fixtures (`test/helpers/agent-browser-harness.ts` contract pages), plus `pushstate`, `vitals`, network route/requests/HAR, diff snapshot/screenshot/url, trace/profiler, console/errors/highlight, stream enable/status/disable, `cookies set --curl`, a `react tree` missing-renderer case, and `wait --download` with explicit assertions around upstream save-path reporting versus on-disk presence. Browser-opening or credential-dependent families such as `inspect`, `dashboard`, `chat`, provider clouds, and OS clipboard flows stay in fake-upstream or manual validation unless a safe deterministic fixture is added; see [`docs/RELEASE.md`](docs/RELEASE.md#real-upstream-contract-validation) for the async download caveat, temp-directory isolation, output-shape fixture maintenance, and common failure triage (including when to refresh `test/fixtures/agent-browser-real-output-shapes.json`).

### `semanticAction` shorthand (top-level tool input)

- **Compilation and validation:** `compileAgentBrowserSemanticAction` in `extensions/agent-browser/lib/input-modes/semantic-action.ts` turns the optional `semanticAction` object into upstream `find` argv; it is mutually exclusive with `args`, `job`, `qa`, `sourceLookup`, `networkSourceLookup`, and `electron` on the same call. Changing allowed actions, locators, or argv shape requires updating the schema beside that helper, the human contract in [`docs/TOOL_CONTRACT.md`](docs/TOOL_CONTRACT.md#semanticaction), examples in [`docs/COMMAND_REFERENCE.md`](docs/COMMAND_REFERENCE.md), and playbook strings in `extensions/agent-browser/lib/playbook.ts` (then `npm run docs -- playbook write`).
- **Optional upstream session field:** optional `semanticAction.session` compiles to a leading `--session <name>` before `find`. `buildExecutionPlan` in `extensions/agent-browser/lib/runtime.ts` then treats the planned argv like any call that already names an upstream session (it does not prepend the extension-managed implicit `--session` on top of that). Unified results echo `details.sessionName` for that name when present with `usedImplicitSession: false` (the call consumed upstream’s named session, not the extension-managed implicit slot); `retry-semantic-action-after-stale-ref` copies `compiledSemanticAction.args`, while `try-*-candidate` actions rebuild role/name fallback argv and preserve the same `--session` prefix.
- **Stale-ref machine follow-up:** for unified failures classified as `stale-ref` when `details.compiledSemanticAction` is present, `extensions/agent-browser/index.ts` appends `retry-semantic-action-after-stale-ref` to any `details.nextActions` from presentation (after `buildAgentBrowserNextActions` in `action-recommendations.ts`). ### Runtime planning (`extensions/agent-browser/lib/runtime.ts`)

This module is **pre-spawn planning only**: it validates raw `args`, derives implicit or fresh managed session names, classifies browser prompts, redacts invocation text, and builds `effectiveArgs` for the upstream binary. It does not spawn processes or touch the filesystem.

- **`buildExecutionPlan`** — Injects `--json`, optionally injects extension-managed `--session` for `sessionMode: "auto"`, applies the ChatGPT headless `--user-agent` workaround when eligible, and returns `validationError` plus a structured `recoveryHint` when launch-scoped flags would be ignored on an already-active implicit session (surfaced to Pi as `details.sessionRecoveryHint` after redaction in `extensions/agent-browser/index.ts`).
- **Command discovery** — `extensions/agent-browser/lib/argv-descriptor.ts` and `extensions/agent-browser/lib/argv-grammar.ts` own `findCommandStartIndex`, `VALUE_FLAGS`, `parseArgvDescriptor`, and `parseCommandInfo`; `buildExecutionPlan` consumes the descriptor instead of re-parsing tokens. `findCommandStartIndex` walks leading tokens using `VALUE_FLAGS` and `GLOBAL_BOOLEAN_FLAGS_WITH_OPTIONAL_VALUES` so known value-taking flags ahead of the upstream command word consume their values before the command is identified. Missing-value prevalidation is narrower: `PREVALIDATED_VALUE_FLAGS` is global-value-only so command-scoped flags or literal free text such as `fill #field --password` are left to upstream parsing instead of being rejected by wrapper heuristics. A smaller set of global booleans may be followed by an optional `true`/`false` literal; when present, that literal is consumed before command discovery continues. When upstream adds new globals that take a separate argv value before the subcommand, extend command discovery and prevalidation; for new command-specific flags, extend command discovery/redaction only when wrapper features need to understand them.
- **Launch-scoped flags** — `extensions/agent-browser/lib/launch-scoped-flags.ts` owns `LAUNCH_SCOPED_FLAG_DEFINITIONS` (currently `--auto-connect`, `--cdp`, `--enable`, `--executable-path`, `--init-script`, `--device`, `--profile`, `--provider`, `-p`, `--session-name`, `--state`); `getStartupScopedFlags` mirrors that metadata. `--auto-connect` counts only when enabled (bare or `true`); `--state` after a `wait` command is treated as a wait predicate, not launch state, so it does not force `sessionMode: "fresh"` during implicit reuse.
- **Tab-correction signal** — `hasLaunchScopedTabCorrectionFlag` is a subset (`--profile`, `--session-name`, `--state`) for post-open tab re-selection when restored tabs steal focus; it intentionally excludes `--cdp`, `--auto-connect`, and provider selection flags (`-p`, `--provider`, `--device`).
- **Upstream subprocess environment** — `extensions/agent-browser/lib/process.ts` passes through a curated env: always-inherited safe names plus every `AGENT_BROWSER_*` variable and provider-related prefixes (`AGENTCORE_`, `BROWSERBASE_`, `BROWSERLESS_`, `BROWSER_USE_`, `KERNEL_`) alongside `AI_GATEWAY_*` and `XDG_*`; extend that module when new upstream credential env families must reach the child without leaking unrelated parent secrets.
- **Upstream subprocess completion** — `runAgentBrowserProcess` spawns `agent-browser` with `stdio: ["pipe", "pipe", "pipe"]` and must not wait forever on `"close"` when a detached descendant inherits those streams. `watchSpawnedChildCompletion` listens for `exit` and `close`, leaves piped stdio intact for the short post-`exit` grace (`EXIT_STDIO_GRACE_MS`) so normal `close` can still win, then destroys those streams only if the fallback resolves with the observed `exit` code. Windows invocations route through PowerShell and timeout/abort cleanup uses `taskkill /T /F` for the full PowerShell/agent-browser process tree before falling back to the direct child signal. Exit-code precedence is `close` → wrapper timeout (`124`) → post-`exit` fallback → spawn failure (`127`), locked by `resolveSpawnedChildExitCode` and `test/agent-browser.process.test.ts`. Changing that window, precedence, or stream teardown order requires updating [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) subprocess bullets, [`docs/SUPPORT_MATRIX.md`](docs/SUPPORT_MATRIX.md) `RQ-0097`, and the stdio-linger regressions in `test/agent-browser.process.test.ts`.
- **Sessionless inspection/local commands** — `isPlainTextInspectionArgs` recognizes `--help`, `-h`, `--version`, and `-V`: those calls keep the caller’s argv (no `--json`), skip implicit `--session`, and are ignored when `restoreManagedSessionStateFromBranch` walks prior tool rows. `needsManagedSession` in `command-policy.ts` gives the same no-implicit-session ownership to read-only `skills list`, `skills get …`, and `skills path …`, local auth-profile management (`auth save/list/show/delete/remove`), plus local/setup commands (`profiles`, `dashboard start/stop`, `device list`, `doctor`, `install`, `upgrade`, `session list`) and targeted/all local saved-state maintenance (`state list/show/clear --all`, `state clear -a`, `state clear <session-name>`, `state clean --older-than <days>`, `state rename`): `buildExecutionPlan` still prepends `--json` for non-plain-text calls but skips implicit/fresh session injection, so successful results omit managed `sessionName` / `usedImplicitSession`. Browser-backed/context-dependent or incomplete commands such as root `session`, untargeted `state clear`, bare `state clean`, `auth login`, and `state save/load` keep normal managed-session injection. Any new upstream sessionless subcommand only picks up this behavior after it is allowlisted in `needsManagedSession` / `command-policy.ts` and covered in `test/agent-browser.runtime.test.ts`.
- **Regression tests** — `test/agent-browser.runtime.test.ts` locks execution-plan injection, launch-scoped blocking, argv edge cases, and related helpers.

Product-facing behavior and rationale live in [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) under **Session model** / **Launch flags**; agent-visible fields live in [`docs/TOOL_CONTRACT.md`](docs/TOOL_CONTRACT.md) (`sessionMode`, `sessionRecoveryHint`, `effectiveArgs`).

### Agent browser efficiency benchmark

`scripts/agent-browser-efficiency-benchmark.mjs` is a **deterministic accounting benchmark**: it does not shell out to `agent-browser`, does not launch a browser, and does not read or write Pi sessions. It models representative `agent_browser` call shapes (including optional `stdin` for `batch` and top-level `job`, `qa`, `sourceLookup`, or `networkSourceLookup` objects that compile to batch) and synthetic model-visible strings to produce comparable totals: scenario success rate, total tool calls, UTF-8 byte volume of model-visible output, stale-ref failure and recovery counts, artifact success count, distinct failure-category coverage, and summed elapsed-time estimates.

- **Run:** `npm run benchmark:agent-browser` prints a Markdown report (default). `npm run benchmark:agent-browser -- --json` prints the same metrics as JSON suitable for saving as a baseline snapshot.
- **Opt-in JSONL sampling:** `npm run benchmark:agent-browser -- --sample-jsonl path/to/session.jsonl` adds a `jsonlSample` block with UTF-8 byte totals and per-workflow/overall p95 sizes for model-visible `agent_browser` tool-result text (content text only; `details` are used for workflow grouping, not byte counting). This does not change deterministic scenario metrics and is not required for default verify.
- **Gate:** Default `npm run verify` runs the full unit suite (`test/**/*.test.ts`), which includes `test/agent-browser.efficiency-benchmark.test.ts` for scenario and comparison assertions, but it does **not** execute the standalone benchmark script’s JSON accounting pass. `npm run verify -- benchmark` runs `scripts/agent-browser-efficiency-benchmark.mjs --json` once and then that test module alone; it stays opt-in so release-style gates are not coupled to re-running the script on every default verify.
- **Compare:** `npm run benchmark:agent-browser -- --compare path/to/prior.json` diffs a saved JSON report against the current in-repo scenario set. Exit code `1` means regressions (for example higher tool-call or model-visible byte totals, dropped success rate, missing failure categories) or unreadable comparison input; optional `jsonlSample` blocks are ignored for regression checks. See the script’s `usage()` text for full exit codes.
- **Evolving the benchmark:** scenarios and `CURRENT_BENCHMARK_VERSION` are defined in the script. Adding, removing, or renaming scenarios—or bumping the version—requires updating `test/agent-browser.efficiency-benchmark.test.ts` and any prose that cites specific scenario ids.

### Maintainer rebaselining workflow

Use this sequence when upstream ships a new `agent-browser` version or help text changes enough to break the live verifier:

1. Capture real `agent-browser … --help` output from the binary you intend to target, then edit `scripts/agent-browser-capability-baseline.mjs`. That file is import-only metadata; it never shells out to `agent-browser`, so rebaselining stays an explicit maintainer decision.
2. Keep `docs/COMMAND_REFERENCE.md` human prose aligned with the baseline: every inventory token the baseline expects in the doc must appear **outside** the two generated regions (`upstream-baseline`, `capability-token-baseline`). `scripts/verify-command-reference.mjs` strips those generated blocks and fails if any required token is missing from the remaining Markdown.
3. Regenerate the HTML-comment bounded blocks with `npm run docs -- command-reference write`. If you also changed playbook source in `extensions/agent-browser/lib/playbook.ts`, refresh every generated doc fragment in one shot with `npm run docs -- write` (runs playbook drift rewrite plus command-reference rewrite).
4. Before committing, run `npm run docs` (or `npm run docs -- command-reference check`) so checked-in blocks cannot drift from the baseline file.
5. When the targeted `agent-browser` is available on `PATH`, run `npm run verify -- command-reference`. It enforces `agent-browser --version` against the baseline, re-samples each `helpCommands` invocation for expected upstream tokens, and repeats the human-token scan against `docs/COMMAND_REFERENCE.md`.

## Preferred testing workflow

Use an end-to-end interactive `pi` run inside `tmux`.

### Rules

- Use two distinct validation modes and do not mix their assumptions:
  - **Quick isolated mode:** launch `pi --no-extensions -e .` from this repository root. Pi settings and auto-discovered extension sources are intentionally bypassed; use this for checkout-only smoke tests and restart the `pi` process after extension edits instead of treating `/reload` as the validation target.
  - **Configured-source lifecycle mode:** run `npm run verify -- lifecycle` before every publish for deterministic `/reload`, exact `--session-id` relaunch, managed-session continuity, persisted-spill, and Pi failure-patch regression coverage (see [`docs/RELEASE.md`](docs/RELEASE.md#pre-release-checks)). Focused extension harness tests cover Pi `session_tree` branch rehydration and cleanup ownership. The lifecycle harness defaults to Pi model `zai/glm-5.1`, **180000 ms** per-step waits in `scripts/verify-lifecycle.mjs`, and accepts `--model <id>` / `--timeout-ms <ms>` passthrough (for example `npm run verify -- lifecycle --model openai-codex/gpt-5.5:minimal`). During development you can also configure exactly one active source for this extension in Pi settings and launch plain `pi` for manual validation of reload/relaunch behaviors because `/reload` exercises auto-discovered/configured resources.
- For code changes, an isolated `pi --no-extensions -e .` smoke session is a pre-commit requirement; whoever made the changes should validate unpublished checkout behavior there before commit/push (a published install alone is not a substitute).
- When validating real extension behavior, run real Pi/model sessions with the target model, for example `pi --model zai/glm-5.1:high --no-extensions -e ./extensions/agent-browser/index.ts`. Do **not** add `--no-session` to these checks: preserve the Pi JSONL transcript so tool-result semantics, model-visible content, and follow-up behavior can be reviewed. Prefer a disposable `--session-dir <temp-dir>` when isolation is needed, then inspect the saved session JSONL and clean up the temp session directory after recording evidence.
- Before every release, run real-site interactive dogfood in `tmux` with `pi --no-extensions -e .` and the native `agent_browser` tool. Cover at least one simple static site and one real docs/product site, include `qa` or `job`/`batch` when those wrapper surfaces changed, verify refs/navigation/artifacts/session cleanup from the transcript, close sessions, remove screenshots/temp files, and record the outcome in release evidence. Automated localhost, fake-upstream, and package smoke gates are not a substitute for this live-site tmux dogfood. When a smoke prompt is meant to validate this extension rather than external dogfood/QA skill behavior, add `--no-skills` and use the bounded release-smoke wording in [`docs/RELEASE.md`](docs/RELEASE.md#public-sauce-demo-checkout-smoke-prompt); run skill-enabled dogfood separately only when skill routing itself is under test. When dashboard, diagnostic, artifact, recording, scroll, or combobox behavior changed, use the public Grafana checklist in [`docs/RELEASE.md`](docs/RELEASE.md#public-grafana-stress-checklist); do not commit private dogfood/VFR skills or run artifacts.
- For installed-package validation after publish: update the real installed package, then exercise it with only the published package active—either temporarily disable/remove the checkout path and run plain `pi` for lifecycle validation, or use an isolated ephemeral smoke run like `pi --no-extensions -e npm:pi-agent-browser-native@<version>`.
- Use `tmux` via bash commands.
- Do **not** use the pi interactive shell extension for this workflow.
- Drive `pi` like a real user by sending prompts normally.
- The extension blocks direct `agent-browser` bash calls by default to push agents toward the native tool. During this package's own development, the guard is bypassed when the session cwd is this package root (`package.json` name `pi-agent-browser-native`) or when `PI_AGENT_BROWSER_ALLOW_DIRECT_BASH=1` is set, so upstream CLI behavior can be debugged directly.
- After extension changes in configured-source lifecycle mode, `/reload` is the minimum, but a full close-and-relaunch of `pi` is preferred for higher confidence. The automated `npm run verify -- lifecycle` harness creates an isolated `PI_CODING_AGENT_DIR`, configures a temporary package source, drives `pi` in `tmux` with an exact `--session-id`, uses a deterministic fake upstream `agent-browser`, checks `/reload` plus restart with the same session id, and captures transcript artifacts on failure.
- If continuing the same conversation after restart, use `/resume` or an explicit session path/id.
- Resumed/exact-session sessions should reflect the updated configured extension source after restart.

### Practical tmux notes

- Prefer `tmux send-keys ... Enter` for prompt submission.
- Capture larger pane ranges when debugging: `tmux capture-pane -p -S -300 -t <session>:0.0`.
- Clean up tmux sessions after testing.
- Before ending a turn, sweep for and remove repo-local scratch files, project-scoped temp artifacts, and lingering browser sessions created during the work unless the user explicitly asked to keep them.
- Do not overfit testing to `example.com`; use it for smoke checks only, then validate against additional realistic pages and flows. The lifecycle harness intentionally uses a fake upstream browser for deterministic lifecycle assertions and does not replace occasional real-browser manual smoke testing before release.
- Prefer **tmux** over `pi -p` for real-browser checkout and release smokes; print mode may hang without a TTY. Use npm 11 when touching `package-lock.json` (`packageManager` in `package.json`). After real-browser runs, close sessions and sweep lingering browser/tmux temp artifacts. Maintainer environment pitfalls (Pi/npm version skew, lifecycle `/reload` timing, automated pane matchers) are documented in [`docs/RELEASE.md`](docs/RELEASE.md#environment-and-automation-pitfalls).

## Current testing focus

Before calling the first implementation usable, verify at minimum:

- missing-`agent-browser` error path is clear
- native tool invocation works from interactive `pi`
- `open` works
- `snapshot -i` works
- `click` works
- `batch` via stdin works
- `eval --stdin` works
- screenshot attachment works
- implicit session reuse works
- explicit `--session` passthrough works
- configured-source `/reload` picks up changed extension code
- full restart + exact `--session-id` relaunch or `/resume` picks up changed extension code
- when `electron.*`, attached-session diagnostics, or `qa.attached` change: exercise `electron.list` → `electron.launch` → browser commands → `electron.cleanup` on a local Electron app (see [`docs/RELEASE.md`](docs/RELEASE.md) and [`docs/ELECTRON.md`](docs/ELECTRON.md))
